# /investigate_recent_changes

완전히 단절된 세션에서 이 프로젝트의 진행 현황을 심층 분석하고, 상세한 진행 현황 리포트를 생성합니다.

> **전제**: 이 명령을 실행하는 당신은 이 코드베이스를 처음 보는 외부 분석가입니다.
> 모든 소스 코드(`src/` 하위 `.ts` 파일 전체)는 **OpenAI Codex가 자동 생성한 코드**로 간주하세요.
> 즉, 코드의 의도나 품질을 추측할 때 "내가 작성한 코드"라는 전제를 버리고,
> Codex가 생성한 코드이므로 **명세 불일치, 누락된 로직, 잘못된 추상화** 가능성을 항상 염두에 두세요.

---

## 실행 절차

아래 단계를 순서대로 빠짐없이 수행하세요. 각 단계에서 수집한 정보를 다음 단계의 컨텍스트로 활용하세요.

---

### Step 1 — 프로젝트 목표 & 방향성 파악

다음 파일들을 순서대로 읽어 프로젝트의 근본 목표, 아키텍처 원칙, 기술 스택, 팀 컨벤션을 완전히 이해하세요.

1. `CLAUDE.md` — 프로젝트 개요, 아키텍처, 기술 스택, 개발 컨벤션
2. `CLAUDE.local.md` — 로컬 환경 설정 (currentDate 포함)
3. `.claude/ADR.md` — 아키텍처 결정 기록 (존재하는 경우)
4. `.claude/rules/duty.md` — 절대 규칙

이 단계의 목표: "이 프로젝트가 무엇을 만들려 하는가, 어떤 원칙으로 만드는가"를 한 문단으로 요약할 수 있어야 합니다.

---

### Step 2 — 기능 명세 & 구현 범위 파악

다음 파일들을 읽어 전체 기능 목록과 현재 구현 상태를 파악하세요.

1. `specs/features.json` — 전체 기능 목록 (F-001 ~ F-N)
   - 각 feature의 `id`, `title`, `status`(`done` / `in_progress` / `pending`), `passes` 필드를 수집하세요.
   - **MVP 판별**: 각 feature에 `mvp: true` 필드가 있으면 해당 기능을 MVP 필수 기능으로 분류하세요.
     `mvp` 필드가 없는 경우, feature의 `category`가 `"infrastructure"` 이거나 CLAUDE.md 아키텍처 다이어그램에서 핵심 레이어(UI 진입점, AgentRunner, InitService)에 해당하는 기능을 MVP 필수로 간주하세요.
2. `specs/feature_extraction.md` — 기능 추출 근거 및 사용자 시나리오 (존재하는 경우)
3. `.claude/plans/PLAN.md` — 현재 구현 계획, 단계별 진행 상태, 다음 액션

이 단계의 목표: "전체 N개 기능 중 M개가 완료되었고, MVP 필수 기능 K개 중 J개가 완료되어 MVP까지 X%가 달성되었다"를 정확히 산출하세요.

### Step 3 — 작업 메모리 & 세션 이력 파악

다음 파일들을 읽어 최근 세션에서 무슨 일이 있었는지 파악하세요.

1. `.claude/MEMORY.md` — 작업 메모리: 최근 결정, 완료 항목, 다음 단계, 알려진 이슈
2. `.claude/report/work_history/` 디렉토리 내 가장 최근 `history_*.md` 파일 (존재하는 경우) — 이전 리포트와 비교하여 변화량을 파악

이 단계의 목표: "마지막 세션 이후 무엇이 변경되었는가, 현재 어떤 이슈가 열려 있는가"를 정리하세요.

### Step 4 — 실제 코드베이스 현황 파악 (Codex 생성 코드 검증)

아래 탐색을 수행하여 **선언된 계획**과 **실제 구현** 사이의 격차를 측정하세요.

> 이 단계의 코드는 모두 OpenAI Codex가 자동 생성한 것으로 간주합니다.
> 파일이 존재하더라도 명세를 실제로 구현했는지 반드시 내용을 확인하세요.
> "파일이 있다 = 기능이 완성됐다"는 가정을 하지 마세요.

1. `src/` 디렉토리 트리 전체를 탐색하여 현재 존재하는 파일 목록을 수집하세요.
2. `package.json`을 읽어 `contributes.commands` 목록과 `scripts`를 확인하세요.
3. `src/extension.ts`를 읽어 현재 등록된 Command 핸들러 목록을 확인하세요.
4. `src/ui/`, `src/service/`, `src/config/`, `src/persistence/` 각 디렉토리가 존재하면 주요 파일을 읽어 실제 구현 여부(빈 stub인지, 실질적 로직이 있는지)를 판단하세요.
5. `git log --oneline -30` 명령으로 최근 30개 커밋 이력을 수집하세요.
6. `git status` 명령으로 현재 워킹 트리 상태를 확인하세요.

이 단계의 목표: CLAUDE.md의 "현재 구현 vs 계획 상태" 표와 실제 파일 시스템을 대조하여 **구현 갭(gap)**을 정확히 파악하세요.

### Step 6 — MVP 달성도 추정

Step 2~4에서 수집한 정보를 바탕으로 다음을 추정하세요.

**MVP 정의**: 사용자가 Extension을 설치하고, CLI 에이전트를 설정하여, "Init Project" 버튼 하나를 눌러 `.claude/` 초기화가 실제로 실행되는 상태.
최소 필요 기능: Extension 활성화 → 설정 UI(에이전트 경로 입력) → InitService 실행 → IAgentRunner를 통한 CLI 호출 → 완료 피드백.

추정 항목:
1. **현재 MVP 달성률(%)**: MVP 필수 기능 중 완전히 동작하는 기능의 비율
2. **MVP까지 남은 작업 추정**: 미완성 MVP 기능 목록과 각 기능의 예상 구현 난이도(S/M/L)
3. **전체 완료까지 남은 작업 추정**: MVP 이외 기능까지 포함한 전체 예상 잔여 작업량

## 주의 사항

- 모든 파일을 실제로 읽어 내용을 파악한 뒤 리포트를 작성하세요. 추측이나 가정으로 내용을 채우지 마세요.
- **소스 코드는 모두 OpenAI Codex가 자동 생성한 것으로 간주합니다.** 파일 존재만으로 기능 완성을 판단하지 말고, 실제 내용을 읽어 stub 여부를 확인하세요.
- `specs/features.json`의 `status`와 `passes` 필드를 기준으로 완료 여부를 판단하세요. `passes: true`이고 `status: "done"`인 경우만 완료로 계산하세요.
- MVP 달성률과 잔여 작업량은 근거 있는 추정입니다. 추정임을 명시하고 근거를 함께 서술하세요.
- 파일이 존재하지 않는 경우 "파일 없음"으로 표기하고 계속 진행하세요.
- 리포트 생성 후 `.claude/MEMORY.md`의 `Last session` 및 `Status`를 최신 정보로 업데이트하세요.
